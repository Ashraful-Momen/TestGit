# PART 1 — GitHub Actions Complete Guide

## 1. GitHub Actions কী? / What is GitHub Actions?

GitHub Actions follows **Event-Driven Architecture**.

**English:** A workflow runs automatically when an **event** (push, pull_request, schedule, manual click, etc.) happens on your repository.

**বাংলা:** Repository-তে কোনো **event** (push, pull request, schedule, manual button click ইত্যাদি) ঘটলেই workflow স্বয়ংক্রিয়ভাবে run হয়।

### File Location (অবশ্যই এই path-এ রাখতে হবে)

```text
your-repo/
└── .github/
    └── workflows/
        └── any-name.yml
```

| Path | ব্যাখ্যা |
|---|---|
| `.github/` | GitHub-এর বিশেষ folder — workflow সবসময় এর ভিতরে থাকে |
| `workflows/` | সব workflow file এই folder-এ রাখতে হয়, নাহলে GitHub চিনতে পারবে না |
| `any-name.yml` | Workflow file YAML format-এ লেখা হয়, নাম যেকোনো কিছু হতে পারে |

## 2. Basic Structure / মৌলিক গঠন

```yaml
name: My First Workflow

on: [push, pull_request]

jobs:
  my-job-name:
    runs-on: ubuntu-latest
    steps:
      - name: First Step
        run: echo "hello world"

      - name: Multi-line Step
        run: |
          echo "hello world" > hello.txt
          cat hello.txt
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `name: My First Workflow` | Name of the workflow, shown in GitHub Actions tab | Workflow-এর নাম, GitHub-এর Actions tab-এ এই নাম দেখা যাবে |
| `on: [push, pull_request]` | Events that trigger this workflow — runs on every push or pull request | কোন event-এ workflow run হবে — প্রতিটি push বা pull request-এ run হবে |
| `jobs:` | A workflow contains one or more jobs. A job = a group of steps | একটি workflow-এ এক বা একাধিক job থাকে। Job = step-গুলোর সমষ্টি |
| `my-job-name:` | Unique ID of the job (no spaces allowed, use dash) | Job-এর unique id (space ব্যবহার করা যাবে না, dash ব্যবহার করুন) |
| `runs-on: ubuntu-latest` | Operating System of the runner machine | কোন Operating System-এর machine-এ এই job run হবে |
| `steps:` | List of small tasks inside a job | Job-এর ভিতরের ছোট ছোট task-এর তালিকা |
| `- name: First Step` | Name of the step (the `-` makes it a YAML list item) | Step-এর নাম (`-` দিয়ে YAML list item শুরু হয়) |
| `run: echo "hello world"` | Executes a single shell command | একটি shell command execute করবে |
| `run: \|` | The `\|` symbol allows running multiple commands in one step | `\|` symbol দিয়ে একটি step-এ একাধিক command run করা যায় |
| `echo "hello world" > hello.txt` | Writes text into a file | Text-কে file-এ লিখবে |
| `cat hello.txt` | Prints the file content | File-এর content দেখাবে |

## 3. Manual Workflow / ম্যানুয়াল ওয়ার্কফ্লো

**English:** This workflow does NOT run automatically. You run it by clicking the "Run workflow" button in the GitHub Actions tab.

**বাংলা:** এই workflow automatically run হয় না। GitHub repo-র **Actions** tab-এ গিয়ে button click করে run করতে হয়।

```yaml
name: Manual Workflow

on:
  workflow_dispatch:

jobs:
  manual-job:
    runs-on: ubuntu-latest
    steps:
      - name: Single Command
        run: echo "hello world"

      - name: Multiple Commands
        run: |
          echo "hello world" > hello.txt
          cat hello.txt
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `name: Manual Workflow` | Name of the workflow | Workflow-এর নাম |
| `on:` | Event section starts | Event অংশ শুরু |
| `workflow_dispatch:` | Manual trigger — a "Run workflow" button appears in the Actions tab | Manual trigger — Actions tab-এ একটি "Run workflow" button তৈরি হবে, click করলে run হবে |
| `manual-job:` | Job ID | Job-এর id |
| `runs-on: ubuntu-latest` | One job runs on ONE operating system | একটি job একটি OS-এ run হয় (একাধিক OS চাইলে matrix ব্যবহার করতে হয় — section 9 দেখুন) |

## 4. Scheduled Workflow (Cron) / সময় অনুযায়ী চলা ওয়ার্কফ্লো

**English:** Runs automatically on a schedule using cron syntax (UTC timezone).

**বাংলা:** নির্দিষ্ট সময় অনুযায়ী স্বয়ংক্রিয়ভাবে run হয় (cron syntax ব্যবহার করে, UTC time-এ)।

```yaml
name: Scheduled Workflow

on:
  schedule:
    - cron: '*/5 * * * *'

jobs:
  cron-job:
    runs-on: ubuntu-latest
    steps:
      - name: Run Task
        run: |
          echo "Running scheduled task"
          date
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `schedule:` | Schedule event — workflow runs by time, not by push | Schedule event — push ছাড়াই নির্দিষ্ট সময়ে workflow run হবে |
| `- cron: '*/5 * * * *'` | Cron expression — runs every 5 minutes | Cron expression — প্রতি ৫ মিনিট পরপর run হবে |
| `date` | Prints current date/time in the log | Log-এ বর্তমান তারিখ ও সময় দেখাবে |

### Cron Syntax — সম্পূর্ণ ব্যাখ্যা (যাতে ইচ্ছেমতো time set করতে পারেন)

```text
┌──────── Minute       (0 - 59)   → কোন মিনিটে
│ ┌────── Hour         (0 - 23)   → কোন ঘণ্টায় (24-hour format)
│ │ ┌──── Day of Month (1 - 31)   → মাসের কোন তারিখে
│ │ │ ┌── Month        (1 - 12)   → কোন মাসে
│ │ │ │ ┌ Weekday      (0 - 6)    → সপ্তাহের কোন দিনে (0 = রবিবার, 1 = সোমবার ... 6 = শনিবার)
│ │ │ │ │
* * * * *
```

### ৫টি Position-এ কী কী Symbol ব্যবহার করা যায়

| Symbol | English | বাংলা | উদাহরণ |
|---|---|---|---|
| `*` | Every value | প্রতিটি value (সবসময়) | Minute-এ `*` = প্রতি মিনিটে |
| `*/n` | Every n units | প্রতি n-এর ব্যবধানে | `*/5` = প্রতি ৫ মিনিটে |
| `n` | Exact value | নির্দিষ্ট value | `30` = ঠিক ৩০ মিনিটে |
| `n,m` | Multiple values | একাধিক নির্দিষ্ট value | `1,15` = ১ তারিখ ও ১৫ তারিখে |
| `n-m` | Range | একটি range-এর মধ্যে | `1-5` = সোমবার থেকে শুক্রবার |

### Practical Examples — নিজে বুঝে time set করুন

| Cron | বাংলা ব্যাখ্যা |
|---|---|
| `*/5 * * * *` | প্রতি ৫ মিনিট পরপর |
| `*/15 * * * *` | প্রতি ১৫ মিনিট পরপর |
| `0 * * * *` | প্রতি ঘণ্টায় (ঠিক :00 মিনিটে) |
| `30 * * * *` | প্রতি ঘণ্টার :30 মিনিটে |
| `0 9 * * *` | প্রতিদিন সকাল ৯টায় (UTC) |
| `30 8 * * *` | প্রতিদিন সকাল ৮:৩০-এ |
| `0 9,18 * * *` | প্রতিদিন সকাল ৯টা ও সন্ধ্যা ৬টায় |
| `0 */6 * * *` | প্রতি ৬ ঘণ্টা পরপর |
| `0 9 * * 1` | প্রতি সোমবার সকাল ৯টায় |
| `0 9 * * 1-5` | প্রতি সোম–শুক্র সকাল ৯টায় (শুধু office day) |
| `0 0 * * 0` | প্রতি রবিবার রাত ১২টায় |
| `0 0 1 * *` | প্রতি মাসের ১ তারিখ রাত ১২টায় |
| `0 0 1,15 * *` | প্রতি মাসের ১ ও ১৫ তারিখে রাত ১২টায় |
| `0 0 1 1 *` | প্রতি বছর ১ জানুয়ারি রাত ১২টায় |
| `0 9 15 6 *` | প্রতি বছর ১৫ জুন সকাল ৯টায় |

### ⚠️ গুরুত্বপূর্ণ — UTC Timezone!

**English:** GitHub Actions cron runs in **UTC** time, NOT your local time. Convert your local time to UTC before writing cron.

**বাংলা:** GitHub Actions-এর cron সবসময় **UTC** time-এ চলে, আপনার local time-এ নয়। তাই cron লেখার আগে local time-কে UTC-তে convert করুন।

```text
বাংলাদেশ (UTC+6) থেকে উদাহরণ:
─────────────────────────────────────────
রাত ১২টা (BD) চাইলে  →  UTC: 18:00  →  cron: '0 18 * * *'
সকাল ৯টা (BD) চাইলে  →  UTC: 03:00  →  cron: '0 3 * * *'
ভোর ৬টা (BD) চাইলে   →  UTC: 00:00  →  cron: '0 0 * * *'
```

**Formula:** `UTC time = Bangladesh time − 6 ঘণ্টা`

**নিজে Time Set করার নিয়ম (Step by Step):**

1. কখন run করাতে চান ঠিক করুন (যেমন: প্রতিদিন রাত ১১:৩০ BD time)
2. UTC-তে convert করুন (১১:৩০ − ৬ ঘণ্টা = ১৭:৩০ UTC)
3. Position অনুযায়ী বসান → minute=`30`, hour=`17`, বাকি সব `*`
4. Result: `- cron: '30 17 * * *'`

## 5. Dependent Jobs (`needs:`) / এক job অন্য job-এর উপর নির্ভরশীল

**English:** Multiple jobs in ONE workflow. Job 2 waits until Job 1 succeeds — using `needs:`.

**বাংলা:** একটি workflow-এর ভিতরে একাধিক job থাকতে পারে। `needs:` দিয়ে বলা যায় — আগের job success না হলে পরের job run হবে না।

```yaml
name: Dependent Jobs Workflow

on:
  push:
    branches:
      - main

jobs:
  first-job:
    runs-on: ubuntu-latest
    steps:
      - name: First Step
        run: |
          echo "hello world" > hello.txt
          cat hello.txt

  second-job:
    runs-on: ubuntu-latest
    needs: [first-job]
    steps:
      - name: Second Step
        run: echo "First job done, now running second job"

  third-job:
    runs-on: ubuntu-latest
    needs: [first-job, second-job]
    steps:
      - name: Final Step
        run: echo "All previous jobs completed!"
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `on: push:` | Runs when code is pushed | Code push করলে run হবে |
| `branches:` | Filter — only specific branches will trigger | Filter — শুধু নির্দিষ্ট branch-এ push হলে trigger হবে |
| `- main` | Only the main branch | শুধু main branch-এ push হলে run হবে |
| `first-job:` | First job ID (all jobs stay inside ONE `jobs:` block) | প্রথম job-এর id (সব job একটি `jobs:` block-এর ভিতরে থাকে) |
| `needs: [first-job]` | This job waits — runs only if first-job succeeds | এই job অপেক্ষা করবে — first-job success হলেই কেবল run হবে |
| `needs: [first-job, second-job]` | A job can depend on multiple jobs | একটি job একাধিক job-এর উপর depend করতে পারে — দুটো success হলেই run হবে |

## 6. Conditional Workflow (path filter) / শর্ত অনুযায়ী ওয়ার্কফ্লো

**Use Case:** Backend folder-এ change হলে শুধু backend pipeline run হবে; frontend change হলে frontend pipeline। যেমন: backend image update হলেই শুধু backend build।

```yaml
name: Conditional Workflow

on:
  push:
    paths:
      - 'backend/**'

jobs:
  backend-job:
    runs-on: ubuntu-latest
    steps:
      - name: Build Backend Image
        run: echo "Backend Image is building..."
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `paths:` | Path filter — workflow runs only when these files/folders change | Path filter — শুধু এই file/folder-এ change হলে workflow run হবে |
| `- 'backend/**'` | Any change inside the backend folder (`**` = all subfolders and files) | backend folder-এর ভিতরের যেকোনো change (`**` = সব subfolder ও file) |
| `run: echo "Backend Image is building..."` | Prints a message (here you would run your real build command) | Message দেখাবে (এখানে আপনি আসল build command লিখবেন) |

### Step-level condition (`if:`) / Step-এর উপরও শর্ত দেওয়া যায়

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Only on main branch
        if: github.ref == 'refs/heads/main'
        run: echo "Deploying from main..."
```

| Line | English | বাংলা |
|---|---|---|
| `if: github.ref == 'refs/heads/main'` | Step runs only if current branch is main | Current branch main হলেই কেবল এই step run হবে |
| `github.ref` | Built-in variable holding the current branch reference | Built-in variable — current branch-এর reference ধরে রাখে |

## 7. Value Passing Between Jobs / এক job থেকে অন্য job-এ value পাঠানো

**English:** Job 1 generates a value → Job 2 receives and prints it. Uses `$GITHUB_OUTPUT` + `outputs:` + `needs:`.

**বাংলা:** একটি job value তৈরি করে, অন্য job সেই value ব্যবহার করে। এজন্য ৩টি জিনিস লাগে — `$GITHUB_OUTPUT`, job-level `outputs:`, এবং `needs:`।

```yaml
name: Value Pass Workflow

on:
  workflow_dispatch:
    inputs:
      name:
        description: 'Enter Your Name'
        required: true
        default: 'unnamed'

jobs:
  get-value-job:
    runs-on: ubuntu-latest
    outputs:
      greeting-message: ${{ steps.generate-message.outputs.message }}
    steps:
      - name: Generate greeting message
        id: generate-message
        run: |
          echo "Hello, ${{ github.event.inputs.name }}!"
          echo "message=Hello, ${{ github.event.inputs.name }}!" >> $GITHUB_OUTPUT

  print-value-job:
    runs-on: ubuntu-latest
    needs: [get-value-job]
    steps:
      - name: Print the greeting message
        run: |
          echo "${{ needs.get-value-job.outputs.greeting-message }}"
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `inputs:` | When run manually, asks the user for input values | Manual run-এর সময় user-এর কাছ থেকে input নেবে |
| `name:` | Name of the input field | Input field-এর নাম |
| `description:` | Hint text shown beside the input box | Input box-এর পাশে দেখানো hint text |
| `required: true` | User must fill this field | User অবশ্যই এই field পূরণ করবে |
| `default: 'unnamed'` | Value used if user enters nothing | User কিছু না লিখলে এই default value ব্যবহার হবে |
| `outputs:` | Declares which values this job will share with other jobs | Declare করে এই job কোন value অন্য job-এর সাথে share করবে |
| `greeting-message: ${{ steps.generate-message.outputs.message }}` | Output name ← value from step. Syntax: `steps.<step-id>.outputs.<variable-name>` | Output-এর নাম ← step থেকে আসা value। Syntax: `steps.<step-id>.outputs.<variable-name>` |
| `id: generate-message` | Gives the step an ID so its output can be accessed later | Step-কে একটি id দেওয়া হলো যাতে পরে এর output access করা যায় |
| `echo "message=..." >> $GITHUB_OUTPUT` | Writes `message=<value>` to the special output file — this creates the step output | `message=<value>` বিশেষ output file-এ লেখা হলো — এটিই step output তৈরি করে। `message=` অংশটাই variable name |
| `${{ github.event.inputs.name }}` | Reads the manual input value typed by the user | User-এর লেখা manual input value পড়বে |
| `needs: [get-value-job]` | Runs only after get-value-job succeeds | get-value-job success হলেই কেবল run হবে |
| `${{ needs.get-value-job.outputs.greeting-message }}` | Reads the other job's output. Syntax: `needs.<job-id>.outputs.<output-name>` | অন্য job-এর output পড়বে। Syntax: `needs.<job-id>.outputs.<output-name>` |

## 8. Predefined Actions (Marketplace) / তৈরি Action ব্যবহার

**English:** Instead of installing tools manually, use ready-made actions from GitHub Marketplace.

**বাংলা:** নিজে manually tool install না করে, GitHub Marketplace-এর ready-made action ব্যবহার করা যায়। যেমন: `actions/setup-node` দিয়ে এক লাইনে Node.js setup।

```yaml
name: Predefined Action Workflow

on:
  workflow_dispatch:

jobs:
  predefined-workflow:
    runs-on: ubuntu-latest
    steps:
      - name: Node.js Version (before setup)
        run: node -v || echo "Node not installed"

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Node.js Version (after setup)
        run: node -v
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `run: node -v \|\| echo "Node not installed"` | Checks Node version; if not installed, prints a message (`\|\|` = run next command if first fails) | Node version check করবে; install না থাকলে message দেখাবে (`\|\|` = প্রথম command fail করলে পরেরটি run হবে) |
| `uses: actions/setup-node@v4` | Uses a pre-built marketplace action to install Node.js | Marketplace-এর তৈরি action ব্যবহার করে Node.js install করবে |
| `with:` | Input parameters for that action | ওই action-এর input parameters |
| `node-version: '20'` | Tells the action which Node version to install | Action-কে বলা হলো কোন Node version install করতে হবে |
| `run: node -v` | Now shows v20.x because setup-node installed it | এখন v20.x দেখাবে কারণ setup-node install করে দিয়েছে |

### জনপ্রিয় Marketplace Actions

```text
actions/checkout@v4          → repo-র code runner-এ নামানো
actions/setup-node@v4        → Node.js setup
actions/setup-python@v5      → Python setup
docker/login-action@v3       → Docker Hub login
docker/build-push-action@v6  → Docker image build + push
appleboy/ssh-action@v1.2.0   → SSH দিয়ে remote server-এ command চালানো
```

**Syntax:** `owner/repository@version` → যেমন `actions/setup-node@v4` মানে github.com/actions/setup-node repo-র v4 version।

### Self-Hosted Runner / নিজের server-এ job চালানো

**English:** By default jobs run on GitHub's machines. But you can run them on your OWN server (self-hosted runner) — useful when you need special hardware, private network access, or want to avoid GitHub's minute limits.

**বাংলা:** Default-এ job GitHub-এর machine-এ run হয়। কিন্তু চাইলে নিজের server-এও job run করতে পারেন (self-hosted runner) — বিশেষ hardware, private network access দরকার হলে, বা GitHub-এর free minute limit বাঁচাতে এটি কাজে লাগে।

```yaml
jobs:
  my-job:
    runs-on: self-hosted
    steps:
      - run: echo "Running on my own server"
```

| Line | English | বাংলা |
|---|---|---|
| `runs-on: self-hosted` | Job runs on your own registered server instead of GitHub's cloud | Job GitHub-এর cloud-এর বদলে আপনার নিজের registered server-এ run হবে |

**Setup করতে:** Repo → Settings → Actions → Runners → New self-hosted runner → GitHub-এর দেওয়া command-গুলো নিজের server-এ run করুন।

## 9. Matrix Workflow (Parallel Jobs) / প্যারালেল ওয়ার্কফ্লো

**English:** Run the SAME job multiple times in parallel with different combinations. Use case: test on multiple OS/Node versions, build multiple Docker images, deploy to multiple servers.

**বাংলা:** একই job বিভিন্ন combination দিয়ে parallel-এ অনেকবার run করা। Use case: একাধিক OS/Node version-এ test, একাধিক Docker image build, একাধিক server-এ deploy।

```yaml
name: Matrix Workflow

on:
  workflow_dispatch:

jobs:
  matrix-job:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        uppercase: [A, B, C]
        lowercase: [a, b, c]
    steps:
      - name: Echo Message
        run: echo "Running Matrix Workflow"

      - name: Print Matrix Values
        run: |
          echo "Uppercase: ${{ matrix.uppercase }}"
          echo "Lowercase: ${{ matrix.lowercase }}"
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `strategy:` | Strategy section — controls HOW the job repeats | Strategy অংশ — job কীভাবে repeat হবে তা নিয়ন্ত্রণ করে |
| `matrix:` | Defines variable lists — the job runs once for every combination | Variable-এর list define করে — প্রতিটি combination-এর জন্য job একবার করে run হয় |
| `uppercase: [A, B, C]` | First variable with 3 values | প্রথম variable — ৩টি value |
| `lowercase: [a, b, c]` | Second variable with 3 values | দ্বিতীয় variable — ৩টি value |
| | Total runs = 3 × 3 = **9 parallel jobs** | মোট run = 3 × 3 = **৯টি parallel job** |
| `${{ matrix.uppercase }}` | Current combination's uppercase value (A or B or C) | বর্তমান combination-এর uppercase value (A অথবা B অথবা C) |
| `${{ matrix.lowercase }}` | Current combination's lowercase value (a or b or c) | বর্তমান combination-এর lowercase value (a অথবা b অথবা c) |

### Practical Example — Multiple OS + Node Version

```yaml
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node: [18, 20, 22]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm test
```

| Line | English | বাংলা |
|---|---|---|
| `runs-on: ${{ matrix.os }}` | OS comes from the matrix — each parallel job uses a different OS | OS matrix থেকে আসছে — প্রতিটি parallel job আলাদা OS ব্যবহার করবে |
| `os: [ubuntu-latest, windows-latest, macos-latest]` | 3 operating systems | ৩টি operating system |
| `node: [18, 20, 22]` | 3 Node versions | ৩টি Node version |
| | Total = 3 OS × 3 Node = **9 test runs** | মোট = 3 OS × 3 Node = **৯টি test run** |
| `node-version: ${{ matrix.node }}` | Installs the Node version of the current combination | বর্তমান combination-এর Node version install করবে |
| `run: npm test` | Runs tests on every OS + Node combination | প্রতিটি OS + Node combination-এ test run করবে |

## 10. Running Scripts From Repo / Repository-র script run করা

**English:** To run any script (Python, Bash, JS) from your repo, you MUST first checkout the code using `actions/checkout`.

**বাংলা:** Repo-র ভিতরের কোনো script (Python, Bash, JS) run করতে হলে **অবশ্যই আগে** `actions/checkout@v4` দিয়ে code runner-এ নামাতে হবে। Checkout ছাড়া runner-এ আপনার file থাকবে না!

```yaml
name: Script Workflow

on:
  workflow_dispatch:

jobs:
  script-job:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Run Python Script
        run: python script.py

      - name: Run Bash Script
        run: bash deploy.sh
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `uses: actions/checkout@v4` | Downloads your repository's code onto the runner machine | আপনার repository-র code runner machine-এ download করবে — এটি ছাড়া runner-এ আপনার কোনো file থাকবে না |
| `run: python script.py` | Runs script.py which exists in your repo | Repo-তে থাকা script.py run করবে |
| `run: bash deploy.sh` | Runs deploy.sh shell script from your repo | Repo-তে থাকা deploy.sh shell script run করবে |

## 11. Secrets & Variables / গোপন তথ্য ব্যবস্থাপনা

**English:** Store credentials (Docker password, AWS keys, SSH keys) in GitHub Secrets — never hardcode them in YAML.

**বাংলা:** Password, API key, SSH key — এসব কখনোই YAML file-এ সরাসরি লিখবেন না। GitHub-এর Secrets-এ রাখুন।

### কোথায় Set করবেন

```text
GitHub → Repository → Settings → Secrets and variables → Actions
    ├── Secrets   → গোপন তথ্য (password, token, key) — encrypted
    └── Variables → সাধারণ config value (non-sensitive)
```

**✅ Best Practice:** Secret/Variable name সবসময় UPPERCASE-এ লিখুন — যেমন `DOCKER_PASSWORD`

```yaml
name: Secrets Workflow

on:
  workflow_dispatch:

jobs:
  secrets-job:
    runs-on: ubuntu-latest
    steps:
      - name: Login to Docker Hub
        run: |
          echo "${{ secrets.DOCKER_PASSWORD }}" | docker login \
            -u "${{ secrets.DOCKER_USERNAME }}" \
            --password-stdin

      - name: Use Variable
        run: echo "App environment is ${{ vars.ENVIRONMENT_NAME }}"
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `echo "${{ secrets.DOCKER_PASSWORD }}"` | Reads the secret value named DOCKER_PASSWORD | DOCKER_PASSWORD নামের secret value পড়বে |
| `\| docker login` | Pipes the password into the docker login command (the `\|` shell pipe passes output as input) | Password-কে docker login command-এ পাঠাচ্ছে (shell-এর `\|` pipe output-কে input হিসেবে দেয়) |
| `-u "${{ secrets.DOCKER_USERNAME }}"` | `-u` = username flag for docker login | `-u` = docker login-এর username flag |
| `--password-stdin` | Reads password from stdin instead of command line — password never appears in logs | Command line-এর বদলে stdin থেকে password পড়ে — log-এ password কখনো দেখা যায় না |
| `${{ vars.ENVIRONMENT_NAME }}` | Reads a Variable (non-secret config). Secrets = `secrets.NAME`, Variables = `vars.NAME` | Variable (non-secret config) পড়বে। Secrets = `secrets.NAME`, Variables = `vars.NAME` |

## 12. Reusable Workflow / পুনঃব্যবহারযোগ্য ওয়ার্কফ্লো

**English:** Write a workflow once, call it from many other workflows.

**বাংলা:** একটি workflow একবার লিখে, অনেক workflow থেকে call করা যায়।

### Step 1: Reusable Workflow তৈরি (`.github/workflows/reusable-build.yml`)

```yaml
name: Reusable Build Workflow

on:
  workflow_call:
    inputs:
      image-name:
        required: true
        type: string

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Build Image
        run: echo "Building Docker image: ${{ inputs.image-name }}"

      - name: Push Image
        run: echo "Pushing Docker image..."
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `workflow_call:` | Special trigger — makes this workflow callable from OTHER workflows | বিশেষ trigger — এই workflow-কে অন্য workflow থেকে call করার যোগ্য করে তোলে |
| `inputs:` | Values the caller must/can send | Caller থেকে কোন value আসবে তা declare করা |
| `image-name:` | Input field name | Input field-এর নাম |
| `required: true` | Caller must provide this value | Caller অবশ্যই এই value দেবে |
| `type: string` | Data type of the input (string/number/boolean) | Input-এর data type (string/number/boolean) |
| `${{ inputs.image-name }}` | Uses the value sent by the caller | Caller-এর পাঠানো value ব্যবহার করবে |

### Step 2: অন্য workflow থেকে Call করা (`.github/workflows/deploy.yml`)

```yaml
name: Use Reusable Workflow

on:
  push:
    paths:
      - "backend/**"

jobs:
  call-reusable-workflow:
    uses: YOUR_USERNAME/YOUR_REPO/.github/workflows/reusable-build.yml@main
    with:
      image-name: backend-api
    secrets: inherit
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `uses: ...reusable-build.yml@main` | Calls the reusable workflow at JOB level — no `runs-on` needed here because the called workflow has its own | Reusable workflow-কে **JOB level**-এ call করা হলো — এখানে `runs-on` লাগে না কারণ call হওয়া workflow-এর নিজের runner আছে। `@main` = repo-র main branch থেকে নেবে |
| `with:` | Sends input values to the reusable workflow | Reusable workflow-কে input value পাঠাচ্ছে |
| `image-name: backend-api` | Value for the `image-name` input | `image-name` input-এর value |
| `secrets: inherit` | Passes all of the caller's secrets to the reusable workflow | Caller-এর সব secrets reusable workflow-এ পাঠিয়ে দেবে |

## 13. Composite Action / নিজের Action তৈরি

**English:** Bundle multiple steps into ONE reusable action that anyone can use — even from other repositories.

**বাংলা:** অনেকগুলো step একত্র করে একটি reusable action বানানো, যা অন্য repo থেকেও ব্যবহার করা যায়।

### Step 1: Composite Action file তৈরি (`action.yml`)

```yaml
name: 'Greeting Action'
description: 'Generates a greeting message'

inputs:
  name:
    description: 'Name to greet'
    required: true
    default: 'unnamed'

outputs:
  message:
    description: 'The greeting message'
    value: ${{ steps.greeting.outputs.message }}

runs:
  using: composite
  steps:
    - name: Generate Greeting
      id: greeting
      shell: bash
      run: |
        echo "message=Hello, ${{ inputs.name }}!" >> $GITHUB_OUTPUT
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `name: 'Greeting Action'` | Name of the action | Action-এর নাম |
| `description:` | What the action does (shown in Marketplace) | Action কী করে তা (Marketplace-এ দেখা যায়) |
| `inputs:` | Parameters the action accepts (optional) | Action-এর গ্রহণযোগ্য parameters (optional) |
| `required: true` / `default:` | Whether the input is mandatory, and fallback value | Input বাধ্যতামূলক কিনা, এবং default value |
| `outputs:` | Values the action returns to the caller | Action caller-কে কী value ফেরত দেবে |
| `value: ${{ steps.greeting.outputs.message }}` | Output comes from the step with id `greeting` | Output আসবে `greeting` id-ওয়ালা step থেকে |
| `runs:` | Execution section starts | Execution অংশ শুরু |
| `using: composite` | Declares this is a composite action (multiple steps bundled) | Declare করা হলো এটি একটি composite action (একাধিক step একসাথে) |
| `id: greeting` | Step ID — used to access its output | Step-এর id — এর output access করতে ব্যবহৃত হয় |
| `shell: bash` | Shell to use — **mandatory** for every `run` step inside a composite action | কোন shell ব্যবহার হবে — composite action-এ প্রতিটি `run` step-এ এটি **বাধ্যতামূলক** |
| `echo "message=..." >> $GITHUB_OUTPUT` | Creates the step output named `message` | `message` নামে step output তৈরি করবে |
| `${{ inputs.name }}` | Reads the action's input value | Action-এর input value পড়বে |

**Publish করতে:** Repo → Releases → Create release → Tag (যেমন `v1`) → Publish। তাহলে অন্যরা `username/repo@v1` দিয়ে ব্যবহার করতে পারবে।

### Step 2: Composite Action ব্যবহার (calling workflow)

```yaml
name: Composite Action Caller

on:
  workflow_dispatch:
    inputs:
      name:
        description: 'Enter Your Name'
        required: true
        default: 'unnamed'

jobs:
  composite-action:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Run Composite Action
        id: greet
        uses: ./
        with:
          name: ${{ github.event.inputs.name }}

      - name: Print Message
        run: |
          echo "The greeting msg is: ${{ steps.greet.outputs.message }}"
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `uses: actions/checkout@v4` | Downloads repo code (needed to find the local action.yml) | Repo code download করবে (local action.yml খুঁজে পেতে দরকার) |
| `id: greet` | ID of this step — used to read the action's outputs | এই step-এর id — action-এর output পড়তে ব্যবহৃত হবে |
| `uses: ./` | Runs the action.yml from the current repo's root folder | Current repo-র root folder-এর action.yml run করবে |
| `with: name: ...` | Passes the manual input into the action | Manual input-কে action-এর ভিতরে পাঠাচ্ছে |
| `${{ steps.greet.outputs.message }}` | Reads the `message` output of the step with id `greet` | `greet` id-ওয়ালা step-এর `message` output পড়বে |

### Practical Example — Laravel + React Setup Action

```yaml
name: 'Laravel React Setup'
description: 'Sets up PHP and Node.js, installs dependencies, and runs tests'

inputs:
  php-version:
    description: 'PHP version to use'
    required: true
    default: '8.2'
  node-version:
    description: 'Node.js version to use'
    required: true
    default: '20'

outputs:
  test-status:
    description: 'Status of the test run'
    value: ${{ steps.run-tests.outputs.test-status }}

runs:
  using: 'composite'
  steps:
    - name: Setup PHP
      uses: shivammathur/setup-php@v2
      with:
        php-version: ${{ inputs.php-version }}

    - name: Install PHP Dependencies
      run: composer install --prefer-dist --no-progress
      shell: bash

    - name: Setup Node.js
      uses: actions/setup-node@v4
      with:
        node-version: ${{ inputs.node-version }}

    - name: Install Node.js Dependencies
      run: npm install
      shell: bash

    - name: Run Laravel Tests
      id: run-tests
      run: |
        php artisan test
        echo "test-status=success" >> $GITHUB_OUTPUT
      shell: bash
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `uses: shivammathur/setup-php@v2` | Popular marketplace action that installs PHP | জনপ্রিয় marketplace action — PHP install করে |
| `php-version: ${{ inputs.php-version }}` | Passes the caller's chosen PHP version | Caller-এর পছন্দের PHP version পাঠাচ্ছে |
| `run: composer install --prefer-dist --no-progress` | Installs Laravel's PHP dependencies (`--prefer-dist` = faster download, `--no-progress` = clean logs) | Laravel-এর PHP dependency install করবে (`--prefer-dist` = দ্রুত download, `--no-progress` = পরিষ্কার log) |
| `shell: bash` | Required in composite actions — which shell runs the command | Composite action-এ বাধ্যতামূলক — কোন shell command চালাবে |
| `run: npm install` | Installs React/front-end dependencies | React/front-end dependency install করবে |
| `id: run-tests` | Step ID — its output becomes the action's output | Step-এর id — এর output-ই action-এর output হবে |
| `php artisan test` | Runs Laravel's test suite | Laravel-এর test চালাবে |
| `echo "test-status=success" >> $GITHUB_OUTPUT` | Creates output `test-status` if tests pass | Test pass হলে `test-status` নামে output তৈরি করবে |

> **Note:** Composite action-এর ভিতরে `actions/checkout` ব্যবহার না করাই ভালো — caller workflow নিজেই checkout করে। তাহলে action সব জায়গায় reusable থাকে।

### Reusable Workflow vs Composite Action

| বিষয় | Reusable Workflow | Composite Action |
|---|---|---|
| Trigger | `workflow_call` | `uses:` step হিসেবে |
| Call হয় | **Job level**-এ | **Step level**-এ |
| Runner | নিজের `runs-on` থাকে | Caller-এর runner-এ চলে |
| ভালো কখন | পুরো pipeline reuse | কয়েকটা step bundle করা |

# PART 2 — Real Production CI/CD Pipeline
## GitHub Actions → Docker Hub → Private VPS

## 1. CI/CD কী?

### CI (Continuous Integration)

Developer যখন GitHub-এ নতুন কোড Push করে, তখন স্বয়ংক্রিয়ভাবে—

- Code Checkout হয়
- Dependencies Install হয়
- Test Run হয়
- Docker Image Build হয়

যদি Test Fail করে তাহলে Pipeline এখানেই Stop হয়ে যায়।

### CD (Continuous Deployment)

যদি CI Success হয় তাহলে—

- Docker Image Docker Hub-এ Push হবে
- VPS-এ SSH Login হবে
- Latest Image Pull হবে
- Old Container Remove হবে
- New Container Run হবে

```text
Developer
```

## 4. VPS Setup

### Step 4.1 — Docker Install (VPS-এ)


| Command | ব্যাখ্যা |
|---|---|
| `sudo apt update` | Package list update করে |
| `sudo apt install docker.io -y` | Docker install করে (`-y` = সব প্রশ্নে yes) |
| `docker --version` | Install ঠিকমতো হয়েছে কিনা check |
| `sudo systemctl enable docker` | Server restart হলেও Docker নিজে চালু হবে |
| `sudo systemctl start docker` | এখনই Docker চালু করে |
| `sudo usermod -aG docker $USER` | Current user-কে docker group-এ যোগ করে — `sudo` ছাড়া docker চালানো যায় |

> Logout/Login করুন — নাহলে `docker` group permission কাজ করবে না।

### Step 4.2 — SSH Key Generate

```bash
ssh-keygen -t ed25519 -C "github-actions-deploy"
```

| Part | ব্যাখ্যা |
|---|---|
| `ssh-keygen` | SSH key pair (public + private) তৈরি করে |
| `-t ed25519` | Key-এর ধরন — আধুনিক, ছোট, দ্রুত ও নিরাপদ |
| `-C "..."` | Comment/label — key চিনতে সুবিধা হয় |

**Public Key দেখুন:**

```bash
cat ~/.ssh/id_ed25519.pub
```

**Private Key দেখুন (এটি GitHub Secret-এ যাবে):**

```bash
cat ~/.ssh/id_ed25519
```

**Public Key-কে authorized_keys-এ যোগ করুন:**

```bash
cat ~/.ssh/id_ed25519.pub >> ~/.ssh/authorized_keys
```

**Permissions ঠিক করুন:**

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
```

| Command | ব্যাখ্যা |
|---|---|
| `chmod 700 ~/.ssh` | ssh folder-এ শুধু owner-এর full access |
| `chmod 600 ~/.ssh/authorized_keys` | File-এ শুধু owner read/write করতে পারবে — না হলে SSH কাজ করবে না |

## 5. Application Files

### Dockerfile

```dockerfile
FROM node:20-alpine

WORKDIR /app

COPY package*.json ./

RUN npm ci --omit=dev

COPY . .

EXPOSE 3000

CMD ["npm", "start"]
```

### Line-by-Line ব্যাখ্যা

| Line | English | বাংলা |
|---|---|---|
| `FROM node:20-alpine` | Base image — Node.js 20 on Alpine Linux (very small size) | Base image — Alpine Linux-এ Node.js 20 (size অনেক ছোট) |
| `WORKDIR /app` | Sets the working directory inside the container | Container-এর ভিতরের working directory ঠিক করে |
| `COPY package*.json ./` | Copies package.json + package-lock.json first (for caching) | প্রথমে package file কপি করে (Docker layer cache কাজে লাগে) |
| `RUN npm ci --omit=dev` | Installs only production dependencies from the lock file | Lock file অনুযায়ী শুধু production dependency install করে |
| `COPY . .` | Copies the rest of the project code | বাকি সব project code কপি করে |
| `EXPOSE 3000` | Documents that the app listens on port 3000 | App 3000 port-এ চলে তা জানায় |
| `CMD ["npm", "start"]` | Command that runs when the container starts | Container start হলে এই command চলবে |

**কেন `npm ci`?**
Production-এ `npm ci` সবসময় `npm install` থেকে দ্রুত এবং deterministic — এটি `package-lock.json` হুবহু follow করে।

```text
README.md

```

এতে Image Size ছোট হয় এবং `.env`-এর মতো sensitive file image-এ যায় না।

### Health Check (Dockerfile-এ যোগ করুন)

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD wget --spider http://localhost:3000 || exit 1
```

| Part | ব্যাখ্যা |
|---|---|
| `--interval=30s` | প্রতি ৩০ সেকেন্ডে health check হবে |
| `--timeout=5s` | ৫ সেকেন্ডে response না এলে fail ধরবে |
| `--retries=3` | টানা ৩ বার fail হলে container "unhealthy" mark হবে |
| `wget --spider http://localhost:3000` | App-কে request পাঠিয়ে দেখে জীবিত কিনা |
| `\|\| exit 1` | Request fail হলে error code দেয় |

## 8. 🎯 Complete Production `deploy.yml`

```yaml
name: Node JS CI/CD Pipeline

on:
  push:
    branches:
      - main

jobs:
  # ============ JOB 1: TEST ============
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'

      - name: Install Dependencies
        run: npm ci

      - name: Run Tests
        run: npm test

  # ============ JOB 2: BUILD & PUSH ============
  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Build and Push Docker Image
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true
          tags: |
            ${{ secrets.DOCKER_USERNAME }}/node-api:latest
            ${{ secrets.DOCKER_USERNAME }}/node-api:${{ github.sha }}

  # ============ JOB 3: DEPLOY TO VPS ============
  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to VPS via SSH
        uses: appleboy/ssh-action@v1.2.0
        with:
          host: ${{ secrets.VPS_IP }}
          username: ${{ secrets.VPS_USER }}
          key: ${{ secrets.VPS_SSH_KEY }}
          script: |
            echo "${{ secrets.DOCKER_PASSWORD }}" | docker login \
              -u "${{ secrets.DOCKER_USERNAME }}" --password-stdin

            docker pull ${{ secrets.DOCKER_USERNAME }}/node-api:latest

            docker stop node-api || true
            docker rm node-api || true

            docker run -d \
              --name node-api \
              -p 3000:3000 \
              --restart always \
              --env-file /home/ubuntu/app/.env \
              ${{ secrets.DOCKER_USERNAME }}/node-api:latest

            docker image prune -f
```

### Line-by-Line ব্যাখ্যা

**JOB 1 — TEST:**

| Line | English | বাংলা |
|---|---|---|
| `uses: actions/checkout@v4` | Downloads the repo code onto the runner | Repo-র code runner-এ download করে |
| `cache: 'npm'` | Caches node_modules between runs — pipeline becomes much faster | Run-গুলোর মধ্যে dependency cache রাখে — pipeline অনেক দ্রুত হয় |
| `run: npm ci` | Installs exact dependencies from package-lock.json | package-lock.json অনুযায়ী হুবহু dependency install করে |
| `run: npm test` | Runs the test suite — if it fails, the whole pipeline stops here | Test চালায় — fail করলে পুরো pipeline এখানেই stop |

**JOB 2 — BUILD:**

| Line | English | বাংলা |
|---|---|---|
| `needs: test` | Build runs only if tests pass | Test pass করলেই কেবল build হবে |
| `uses: docker/login-action@v3` | Marketplace action that logs into Docker Hub | Docker Hub-এ login করা marketplace action |
| `username/password: ${{ secrets... }}` | Credentials come from GitHub Secrets — never hardcoded | Credential আসে GitHub Secrets থেকে — কখনো সরাসরি লেখা নয় |
| `uses: docker/build-push-action@v6` | Builds the Docker image AND pushes it to Docker Hub | Docker image build করে Docker Hub-এ push করে |
| `context: .` | Build context = current folder (where the Dockerfile is) | Build context = current folder (Dockerfile যেখানে আছে) |
| `push: true` | Push the image after building | Build-এর পর image push করবে |
| `...node-api:latest` | First tag — always points to the newest image | প্রথম tag — সবসময় সর্বশেষ image-কে নির্দেশ করে |
| `...node-api:${{ github.sha }}` | Second tag — unique per commit, enables rollback | দ্বিতীয় tag — প্রতিটি commit-এ unique, rollback সহজ করে |

**JOB 3 — DEPLOY:**

| Line | English | বাংলা |
|---|---|---|
| `needs: build` | Deploy runs only if the image was pushed successfully | Image push success হলেই কেবল deploy হবে |
| `uses: appleboy/ssh-action@v1.2.0` | Marketplace action that SSHes into a remote server and runs commands | Remote server-এ SSH করে command চালানোর marketplace action |
| `host: ${{ secrets.VPS_IP }}` | IP address of your VPS | আপনার VPS-এর IP address |
| `username: ${{ secrets.VPS_USER }}` | SSH user on the VPS | VPS-এর SSH user |
| `key: ${{ secrets.VPS_SSH_KEY }}` | Private SSH key for authentication | Authentication-এর জন্য private SSH key |
| `script: \|` | All commands below run ON THE VPS via SSH | নিচের সব command VPS-এর ভিতরে SSH দিয়ে run হবে |
| `echo ... \| docker login ... --password-stdin` | Logs into Docker Hub from the VPS (needed to pull private images) | VPS থেকে Docker Hub-এ login করে (private image pull করতে দরকার) |
| `docker pull .../node-api:latest` | Downloads the newest image onto the VPS | VPS-এ সর্বশেষ image download করে |
| `docker stop node-api \|\| true` | Stops the old container. `\|\| true` = don't fail if the container doesn't exist (first deploy) | পুরোনো container বন্ধ করে। `\|\| true` = container না থাকলেও pipeline fail হবে না (প্রথম deploy-এর ক্ষেত্রে) |
| `docker rm node-api \|\| true` | Removes the old container | পুরোনো container মুছে দেয় |
| `docker run -d` | Starts a new container in detached (background) mode | নতুন container background-এ চালু করে |
| `--name node-api` | Names the container (makes stop/rm/logs easy) | Container-এর নাম দেয় (stop/rm/logs সহজ হয়) |
| `-p 3000:3000` | Maps VPS port 3000 → container port 3000 | VPS-এর 3000 port → container-এর 3000 port |
| `--restart always` | Container auto-restarts if it crashes or the VPS reboots | Container crash বা VPS reboot হলেও নিজে আবার চালু হবে |
| `--env-file /home/ubuntu/app/.env` | Loads environment variables from the .env file stored on the VPS (NOT inside the image) | VPS-এ রাখা .env file থেকে environment variable পড়ে (image-এর ভিতরে নয়) — এই file আগে থেকে VPS-এ রাখতে হবে |
| `docker image prune -f` | Deletes old unused images to free disk space | পুরোনো unused image মুছে disk খালি রাখে |

## 10. Rollback Strategy

```text
test → build → deploy
```

যদি Test Fail করে:
- Build ❌ হবে না
- Deploy ❌ হবে না
- Old Container চলতেই থাকবে ✅ (zero downtime)

**পুরোনো version-এ ফিরে যেতে (rollback):**

```bash
# VPS-এ SSH করে:
docker pull username/node-api:<OLD_COMMIT_SHA>
docker stop node-api && docker rm node-api
docker run -d --name node-api -p 3000:3000 --restart always \
  --env-file .env username/node-api:<OLD_COMMIT_SHA>
```

`github.sha` tag ব্যবহার করলে যেকোনো পুরোনো Image-এ মুহূর্তেই Rollback করা যায়।

## 11. Best Practices / সেরা অনুশীলন

- ✅ `.env` কখনোই git-এ commit করবেন না (`.gitignore`-এ রাখুন)
