## 1. Why Version Control?

Imagine you're editing an important document. You make changes, and suddenly everything breaks. Without version control, you're stuck. With version control, every change is a **snapshot** you can return to.

> 💡 **Mental Model (from my notes):** In Linux, we use snapshots to go back to previous system states after changes. **Git does exactly the same thing for your code.**

### The Three Big Ideas

| Concept              | What it means                                                                                                                         |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| **Commit**     | A saved snapshot of your project. 1 commit = 1 stage/snapshot.                                                                        |
| **Checkpoint** | After committing, you can travel**back and forward** between versions of your app.                                              |
| **Branch**     | A parallel timeline — like a tree splitting — so multiple people can work on different features without breaking each other's work. |

### Why production teams can't live without Git

- **Safety:** Nothing is ever truly lost. Every commit is recoverable.
- **Collaboration:** 10 developers can work on the same codebase at once.
- **Accountability:** `git log` tells you *who* changed *what*, *when*, and *why*.
- **Code review:** Changes get reviewed before they touch production.
- **Deployment:** CI/CD pipelines trigger from Git events (push, merge, tag).

### Git vs GitHub — don't confuse them!

- **Git** = the version control *tool* running on your machine (works fully offline).
- **GitHub** = a *hosting service* for Git repositories online (adds pull requests, issues, CI/CD, code review). Alternatives: **GitLab**, **Bitbucket**.

## 2. Installing Git & First-Time Configuration

### Install

- **Windows:** Download from [git-scm.com](https://git-scm.com) — install with **default settings**.
- **Linux (Debian/Kali/Parrot/Ubuntu):**
  ```bash
  sudo apt update && sudo apt install git -y
  ```
- **Mac:**
  ```bash
  brew install git
  ```

Verify:

```bash
git --version
```

### First-time identity setup (required before your first commit)

Git stamps every commit with your name and email. Set it once, globally:

```bash
git config --global user.name "Your Full Name"
git config --global user.email "you@example.com"
```

### Check your configuration

```bash
git config --list
```

### Useful global settings for production work

```bash
git config --global init.defaultBranch main   # new repos start with 'main', not 'master'
git config --global pull.rebase false         # pull = merge (predictable for teams)
git config --global core.editor "code --wait" # use VS Code as commit editor
```

> ⚠️ **Self-hosted GitLab over HTTP (no SSL)?** You may need:
>
> ```bash
> git config http.sslverify false
> ```
>
> Only do this on a trusted internal server — never as a global habit.

### Three config scopes (know the difference)

| Scope  | Flag         | File               | Use case                                             |
| ------ | ------------ | ------------------ | ---------------------------------------------------- |
| System | `--system` | `/etc/gitconfig` | Every user on the machine                            |
| Global | `--global` | `~/.gitconfig`   | You, everywhere (most common)                        |
| Local  | *(none)*   | `.git/config`    | One specific repo — e.g. work email for a work repo |

```bash
# Per-repo identity (overrides global):
cd ~/work/project
git config user.email "you@company.com"
```

## 3. The Core Workflow

Every day of your professional life, you will repeat this cycle:

```
Edit files → git status → git add → git commit → git push
```

### Initialize a repository

```bash
mkdir my-project && cd my-project
git init
```

This creates a hidden `.git/` folder — the entire database of your project's history. Delete it and the history is gone.

### Check status — your most-used command

```bash
git status
```

Shows: current branch, staged files, unstaged changes, untracked files. **Run it constantly.** Professional developers run `git status` dozens of times a day.

### Stage files (add to the "waiting room")

```bash
git add fileName.txt          # one file
git add folderName/           # one folder
git add .                     # everything in the current folder
```

> 🧠 **Concept:** Git has 3 areas:
>
> 1. **Working directory** — where you edit.
> 2. **Staging area (index)** — where `git add` puts changes, ready to be snapshotted.
> 3. **Repository** — where commits live permanently.

### Commit (create the snapshot)

```bash
# Single-line commit:
git commit -m "Add login page"
# (-m means message)

# Multi-line commit (opens editor — title + body):
git commit
```

> ⚠️ **Golden rule:** always `git add` **before** `git commit`. A commit only saves what's *staged*.

### Unstage / untrack a file

```bash
git rm --cached fileName.txt        # stop tracking, keep the file on disk
git restore --staged fileName.txt   # modern way to unstage
```

### The daily loop, end to end

```bash
git status
git add .
git commit -m "Implement user registration form"
git status        # confirm clean
```

## 4. Reading History

```bash
git log                 # full history: hash, author, date, message
git log --oneline       # compact: one line per commit ⭐ (use this most)
git log --graph --oneline --all   # ASCII graph of branches
```

### Pretty-format the log (great for standups/reports)

```bash
git log --pretty=format:"%h - %an, %ar : %s"
# 3f2a1b9 - Ashraful, 2 hours ago : Fix payment webhook
```

| Placeholder | Meaning                      |
| ----------- | ---------------------------- |
| `%h`      | short commit hash            |
| `%an`     | author name                  |
| `%ar`     | relative date ("2 days ago") |
| `%s`      | commit subject               |

### Understand HEAD

- `HEAD` is a **pointer** to your current position — usually the latest commit on your current branch.
- `HEAD~1` = one commit back, `HEAD~2` = two commits back, etc.

### 👁️ Visual tool: Git Graph (VS Code) — highly recommended

1. Install the **Git Graph** extension in VS Code.
2. Press `Ctrl+Shift+P` → search **"Git Graph: View Git Graph"**.
3. Click any commit to see its full details and diff.

> 💡 **From my notes:** Git Graph is extremely useful for **bug hunting** — you can visually find exactly which commit introduced a bug, then click it to inspect the changes.

## 5. The .gitignore File

Some files should **never** enter your repository: secrets, dependencies, build output, IDE settings.

Create a file named `.gitignore` in the repo root:

```gitignore
# files
.env
config/secrets.yml

# folders
/vendor
/node_modules
/AllApiDocumentationForDeveloper/
/ThunderClientApiBackup/
/TaskSchedulingFile/

# patterns
*.log
dist/
```

> 💡 **Pro tip (from my notes):** Don't write `.gitignore` by hand! Go to **[gitignore.io](https://www.toptal.com/developers/gitignore/)**, type your stack (e.g. `django`, `react`, `node`, `laravel`), and it generates a complete, community-maintained `.gitignore` for you.

### ⚠️ The #1 .gitignore trap: already-tracked files

`.gitignore` only stops **new** files from being tracked. If you already committed `.env` or `node_modules/`, adding them to `.gitignore` does **nothing** until you remove them from Git's cache:

```bash
git rm --cached .env
git rm --cached -r node_modules/
git rm --cached -r vendor/
git commit -m "Stop tracking ignored files"
```

### 💣 Nuclear option: rebuild the whole cache

When lots of junk got committed before the `.gitignore` existed:

```bash
git rm -r --cached .
git add .
git commit -m "Rebuild git cache with .gitignore"
```

This untracks **everything**, then re-adds only what `.gitignore` allows. Your files on disk are untouched — only Git's index is rebuilt.

> 🔐 **Security rule:** if a real secret (password, API key) was ever committed, removing it is not enough — the secret lives in history. **Rotate the credential immediately**, then clean history if needed.

## 6. Branching

> 🌳 **Mental model (from my notes):** A branch works like a **tree**. From `main`, you grow sub-branches. Each commit's "parent" is the previous commit's hash — a branch is just a pointer into that chain.

**Why branch?** In a real team, one person builds the homepage, another builds the contact page, another works on payments — each on their own branch. Nobody's half-finished code breaks anyone else's work.

### Branch commands

```bash
git branch                    # list local branches
git branch -a                 # list ALL branches (local + remote)
git branch -v                 # list with last commit on each
git branch --show-current     # just print the current branch name ⭐

git checkout -b feature/login   # create AND switch (classic)
git switch -c feature/login     # create AND switch (modern)

git checkout main               # switch to existing branch
git switch main                 # modern equivalent
```

> 🧠 When you create a branch, the branch you were **on** becomes its parent — the new branch inherits the entire history up to that point.

> ⚠️ **Warning (from my notes):** when you switch from a child (feature) branch back to the parent branch, files that only exist in the child branch **disappear from your working folder** (they're safe in Git, just not checked out). Commit or stash before switching!

### Deleting branches

```bash
git branch -d feature/login    # safe delete (refuses if unmerged)
git branch -D feature/login    # force delete ⚠️
git push origin --delete feature/login   # delete on GitHub/GitLab too
```

### Branch naming conventions (production standard)

```
feature/user-authentication
bugfix/login-redirect-loop
hotfix/payment-crash
release/v2.1.0
```

Lowercase, hyphens, a category prefix. Your future teammates will thank you.

## 7. Merging & Resolving Conflicts

Merging copies all commits from one branch into another.

### Scenario 1: Fast-Forward Merge (the easy one)

You branched `feature` off `main`, committed there, and `main` **hasn't moved** since. Git just slides the `main` pointer forward:

```bash
git switch main
git merge feature
git branch -d feature        # clean up after merging
```

### Scenario 2: Merge Commit (both branches moved)

`main` got new commits *and* `feature` got new commits. Git creates a special **merge commit** that joins the two histories:

```bash
git switch main
git merge feature            # may open an editor for the merge message
```

### Scenario 3: Merge Conflict (same lines edited on both sides)

Git can't decide which version wins, so it stops and asks you. Conflict markers appear in the file:

```
<<<<<<< HEAD
current branch's code
=======
incoming branch's code
>>>>>>> feature
```

**Resolution workflow (memorize this):**

1. `git status` — see which files conflict.
2. Open each file, choose: **Accept Current | Accept Incoming | Accept Both | hand-edit** (VS Code gives you buttons for this).
3. Delete the conflict markers (`<<<<<<<`, `=======`, `>>>>>>>`).
4. `git add .`
5. `git commit -m "Merge feature into main, resolve conflicts"`

> 💡 **From my notes:** if you can't switch or merge because of uncommitted work, do `git add . && git commit -m "wip"` first, then switch/merge. (Or better — use `git stash`, see Chapter 15.)

### Keeping a dev branch updated from main (a real daily task)

> **Real question from my notes:** *There are 2 branches: `momen` and `momen-dev`. All updated code is in `momen`. How do I update `momen-dev`?*

```bash
git switch momen-dev          # 1. go to the branch you want to UPDATE
git merge origin/momen        # 2. merge the source branch into it
# resolve conflicts if any, then:
git push origin momen-dev     # 3. push the updated branch
```

**Rule of direction:** switch to the branch that should *receive* the changes, then merge the branch that *has* the changes.

### ⚠️ The golden rule of merging (from my notes)

> **Before merging, always pull the latest code first**, then resolve conflicts locally. Skipping this is how you get fast-forward errors and rejected pushes.

```bash
git switch main
git pull origin main
git merge feature
```

## 8. Undoing Things Safely

### Undo uncommitted changes

```bash
git restore file.txt          # discard changes in one file
git restore .                 # discard ALL unstaged changes ⚠️
git clean -fd                 # delete untracked files & folders ⚠️ (gone forever!)
```

### Undo the last commit

```bash
# Keep the changes, undo only the commit (SAFE — recommended):
git reset --soft HEAD~1

# Undo commit AND unstage changes:
git reset --mixed HEAD~1      # (this is the default)

# Go back one step and DESTROY the changes (from my notes — dangerous!):
git reset --hard HEAD~1       # ⚠️ changes are gone
```

### Undo a commit that's already been pushed (the SAFE way)

`reset` rewrites history — never do it on shared branches. Instead, `revert` creates a **new commit that undoes** the old one:

```bash
git revert HEAD
git push
```

### Replace one branch's content entirely with another's

> **From my notes:** remove all code from branch `momen` and replace it with branch `hello`'s code:

```bash
git switch momen
git reset --hard hello                 # momen now = hello, exactly
git push --set-upstream origin momen   # may need --force if history diverged
```

### The safety ladder

| Command              | Local changes | Staged    | Commits         | Safe on shared branches? |
| -------------------- | ------------- | --------- | --------------- | ------------------------ |
| `git restore`      | destroyed     | kept      | kept            | ✅ (local only)          |
| `git reset --soft` | kept          | kept      | removed         | ⚠️ only if not pushed  |
| `git reset --hard` | destroyed     | destroyed | removed         | ⚠️ only if not pushed  |
| `git revert`       | —            | —        | new undo commit | ✅**always safe**  |

> 🛟 **Secret safety net:** `git reflog` shows *everywhere HEAD has been*, even after a bad reset. Find the hash before your mistake and `git reset --hard <hash>` to resurrect your work.

## 9. Remote Repositories

A **remote** is a copy of your repo on a server (GitHub, GitLab, Bitbucket, or your company's self-hosted GitLab). `origin` is the default nickname for your main remote.

### Push a brand-new local project to GitHub

```bash
echo "# MyProject" >> README.md
git init
git add README.md
git commit -m "first commit"
git branch -M main                              # rename master → main
git remote add origin https://github.com/YourUser/MyProject.git
git push -u origin main
```

> `-u` (or `--set-upstream`) links your local branch to the remote branch so future `git push` / `git pull` need no arguments. You only need it once per branch.

### Push an existing local repo

```bash
git remote add origin https://github.com/YourUser/MyProject.git
git branch -M main
git push -u origin main
```

### Push other branches

```bash
git push -u origin feature_branch_name
```

### Clone (download a repo)

```bash
git clone https://github.com/user/repo.git
git clone git@github.com:user/repo.git      # SSH version
```

### Pull vs Fetch (know the difference!)

```bash
git pull      # fetch remote changes AND merge them into your branch
git fetch     # download remote changes WITHOUT touching your work
```

> 💡 **From my notes:** when a teammate commits to a shared branch, `git pull` brings their online commits into your offline repo.

**Professional habit:** when you're unsure what teammates pushed, `git fetch` first, inspect with `git log HEAD..origin/main --oneline`, *then* merge.

### Get a branch that exists only on the remote

> **From my notes:** after cloning, `main` comes automatically, but other branches must be fetched manually:

```bash
git fetch origin momen
git checkout -b momen origin/momen     # create local branch tracking the remote one
```

### Inspect & change remotes

```bash
git remote -v                                    # show remote URLs
git remote set-url origin <new-url>              # change where 'origin' points
```

> **Real use case (from my notes):** pointing a repo at a new GitLab server:
>
> ```bash
> git remote set-url origin http://your-gitlab-server/user/project.git
> ```

## 10. SSH Keys

Typing passwords is for amateurs. Production engineers authenticate with **SSH keys**: a private key stays on your machine, the public key goes on GitHub/GitLab.

### Full setup (works on Linux/Kali/Parrot/Mac)

**1. Check for existing keys:**

```bash
ls -al ~/.ssh
# look for id_ed25519 / id_ed25519.pub (modern) or id_rsa / id_rsa.pub (legacy)
```

**2. Generate a key (ed25519 recommended):**

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
# press Enter for default location; set a passphrase for extra security

# legacy systems:
ssh-keygen -t rsa -b 4096 -C "your_email@example.com"
```

**3. Start the SSH agent and load your key:**

```bash
eval "$(ssh-agent bash)"
ssh-add ~/.ssh/id_ed25519
ssh-add -l        # verify it's loaded
```

**4. Copy your PUBLIC key (never the private one!):**

```bash
cat ~/.ssh/id_ed25519.pub
```

**5. Add it to your host:**

- GitHub → Settings → [SSH keys](https://github.com/settings/keys) → New SSH key
- GitLab → Preferences → [SSH Keys](https://gitlab.com/-/profile/keys)
- Bitbucket → Account settings → SSH keys

**6. Test:**

```bash
ssh -T git@github.com
# Hi username! You've successfully authenticated, but GitHub does not provide shell access.
ssh -T git@gitlab.com
```

**7. Clone via SSH:**

```bash
git clone git@github.com:YourUser/repo.git
```

**8. Convert an existing repo from HTTPS to SSH:**

```bash
git remote -v
git remote set-url origin git@github.com:YourUser/repo.git
```

### 💡 Auto-start the SSH agent on login (from my notes)

Add to `~/.bashrc` (or `~/.zshrc`):

```bash
# Auto-start SSH agent
if [ -z "$SSH_AUTH_SOCK" ]; then
  eval "$(ssh-agent bash)"
  ssh-add ~/.ssh/id_ed25519 &>/dev/null
fi
```

Then `source ~/.bashrc`.

### 🏢 Managing MULTIPLE accounts on one machine (work + personal)

Use `~/.ssh/config`:

```
# Personal GitHub
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519

# Work GitHub
Host github-work
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_work
```

Then clone work repos with `git clone git@github-work:company/repo.git`.

## 11. HTTPS, Tokens & Credential Managers

GitHub no longer accepts account passwords over HTTPS — you use a **Personal Access Token (PAT)** as your password.

### Create a token

GitHub → Settings → Developer settings → Personal access tokens:

- Name it descriptively (`"Laptop - work repos"`)
- **Set an expiration** (never "no expiration" for shared machines)
- Scopes: `repo` (private repos), `workflow` (CI/CD)

Copy it immediately — **it's shown only once.**

### Store credentials so you don't retype them

```bash
# Windows:
git config --global credential.helper manager-core
# Mac:
git config --global credential.helper osxkeychain
# Linux (plaintext file — acceptable only on your private machine):
git config --global credential.helper store
```

### Security best practices (from my notes)

- ✅ Prefer **SSH keys over tokens** for long-term use
- ✅ Use **short expiration** for temporary tokens
- ✅ Add **passphrases** to SSH keys
- 🚫 **Never commit tokens or credentials to git** — keep them in `.gitignore`d files
- 🚫 Don't share tokens between machines — one token per machine

## 12. Using Git on Someone Else's PC

Real-world situation: you must run CI/CD or fix something on a colleague's/company PC, then **leave no trace of your credentials**. Here is the complete playbook from my notes.

### Option 1: Personal Access Token (fastest for temporary use)

**Setup:**

1. GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic) → Generate new token
2. Short expiration (e.g. 7 days), scopes: `repo` + `workflow`
3. Copy the token immediately

**On the other PC:**

```bash
git config --global credential.helper store   # or manager-core (Win) / osxkeychain (Mac)
git clone https://github.com/yourusername/repo.git
# paste the TOKEN when asked for a password

# ... do your work ...
git add . && git commit -m "CI/CD fixes" && git push
```

**Cleanup (do ALL of these):**

```bash
# erase stored credentials
git credential-cache exit                 # Linux
git credential-manager erase https://github.com   # Windows
git credential-osxkeychain erase https://github.com  # Mac

rm -f ~/.git-credentials                  # if helper 'store' was used
git config --global --unset user.name
git config --global --unset user.email
git config --global --unset credential.helper
rm -rf ~/repo                             # delete the cloned project
```

**Final step on YOUR main PC:** GitHub → Personal access tokens → **Delete the token.** It's now completely dead.

### Option 2: Temporary SSH key (more secure)

```bash
# 1. Generate a DEDICATED key (custom name so it can't clash with anything):
ssh-keygen -t ed25519 -C "temp-cicd@pc" -f ~/.ssh/id_ed25519_temp -N "passphrase"

# 2. Show the public key and add it to GitHub (Settings → SSH keys → "Temp CI/CD PC"):
cat ~/.ssh/id_ed25519_temp.pub

# 3. Point SSH at this key via ~/.ssh/config:
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_temp

# 4. Test and work:
ssh -T git@github.com
git clone git@github.com:yourusername/repo.git
```

**Cleanup:**

```bash
rm ~/.ssh/id_ed25519_temp ~/.ssh/id_ed25519_temp.pub
# remove the entry from ~/.ssh/config
# delete the key in GitHub → Settings → SSH and GPG keys
```

### Option 3: GitHub CLI (cleanest)

```bash
gh auth login      # authenticate with token
# ... work ...
gh auth logout     # wipes all credentials instantly
```

### ✅ Post-visit security checklist

- [ ] Token deleted on GitHub / SSH key removed
- [ ] Credential manager erased
- [ ] `~/.git-credentials` deleted (if it existed)
- [ ] Git config (name/email) unset
- [ ] Cloned repos deleted
- [ ] Browser sessions logged out (if you used the web UI)

## 13. Pull Requests & Code Review

In production, **nobody pushes directly to `main`.** All code enters through a **Pull Request (PR)** — a proposal to merge one branch into another, with review.

### The standard PR flow

```bash
git switch -c feature/shopping-cart
# ... work ...
git add . && git commit -m "Add shopping cart"
git push -u origin feature/shopping-cart
```

Then on GitHub: **Pull Requests → New Pull Request** → base: `main`, compare: `feature/shopping-cart` → describe your change → request review → teammate approves → **merge**.

After merge, update your local copy:

```bash
git switch main
git pull
git branch -d feature/shopping-cart        # local cleanup
```

### Create PRs from the terminal with GitHub CLI

> **From my notes** — PR from `dev1` (head) into `main` (base):

```bash
gh pr create --base main --head dev1 --title "Title of your PR" --body "Description of your PR"
```

### 🛡️ Branch protection (how real teams enforce review)

Repo owner: **Settings → Branches → Add branch protection rule** for `main`:

- ✅ Require pull request reviews before merging
- ✅ Require status checks to pass (CI must be green)
- ✅ Disallow force pushes

Result: direct pushes to `main` are **blocked**. Every change is reviewed. This is non-negotiable in professional teams.

```
┌───────────────────────┐
│   GitHub Repo         │
│   main (protected) 🔒 │◄──┐  Merge only after review + CI green
└───────────────────────┘   │
                            │ Pull Request
┌───────────────────────┐   │
│  Developer's machine  │───┘
│  feature branch       │
└───────────────────────┘
```

### Writing a good PR

- **Small** — one PR = one concern. 200 lines reviewed well beats 2000 lines rubber-stamped.
- **Descriptive title** — "Fix null pointer in checkout when cart is empty", not "fix bug".
- **Body** — what, why, how to test, screenshots for UI.
- **Respond to review comments** by pushing new commits to the same branch — the PR updates automatically.

## 14. Forking

**Fork** = copy someone else's repo into *your* GitHub account, so you can contribute to projects you don't have write access to (open source!).

### The complete fork workflow

```
┌───────────────────┐     fork      ┌───────────────────┐
│  Original repo    │ ────────────► │  Your copy        │
│  (Profile A)      │               │  (Profile B)      │
└───────────────────┘               └───────────────────┘
        ▲                                    │ clone
        │                            ┌───────────────────┐
        │                            │  Your local repo  │
        │                            │  change → commit  │
        │                            │  → push           │
        └─────── Pull Request ◄──────└───────────────────┘
                (A reviews & merges;
                 you appear in Contributors 🎉)
```

**Part 1 — fork & work:**

1. On GitHub profile A's repo, click **Fork** → a copy appears under your account (B).
2. Clone **your fork**:
   ```bash
   git clone https://github.com/B/contribute_repo.git
   ```
3. Make changes, commit, push to **your fork**:
   ```bash
   git add . && git commit -m "Fix typo in docs"
   git push origin main
   ```

**Part 2 — pull request:**

1. Open your fork on GitHub → **Pull Request** (or "Contribute").
2. Select: base = **A's repo + A's branch** (usually `main`) ← compare = **your repo + your branch**.
3. Submit. Owner A reviews; if merged, your work enters the original project and **you appear in the Contributors list**.

### 💡 Keep your fork updated

```bash
git remote add upstream https://github.com/A/contribute_repo.git
git fetch upstream
git merge upstream/main        # or: git rebase upstream/main
git push origin main
```

> **Fork vs Clone:** *Clone* downloads a repo to your PC. *Fork* copies a repo from someone else's GitHub account into **your** GitHub account. Fork → clone → change → push → pull request.

## 15. Stashing

> **Real scenario (from my notes):** you're mid-task on branch `momen` with uncommitted changes, and someone says *"merge now, it's urgent!"* You can't commit half-done work and can't switch branches with a dirty tree. **Stash it.**

```bash
git stash            # copy current uncommitted work into Git's memory, clean the tree
# ...switch branches, do the urgent merge, come back...
git stash pop        # paste your saved work back and remove it from the stash list
```

### Full stash toolkit

```bash
git stash                          # stash tracked changes
git stash -u                       # include untracked files too
git stash save "wip: cart total"   # named stash (easy to find later)
git stash list                     # see all stashes
git stash show -p stash@{0}        # preview what's inside
git stash apply                    # restore but KEEP the stash entry
git stash pop                      # restore and DELETE the entry
git stash drop stash@{0}           # delete one stash
git stash clear                    # delete all stashes ⚠️
```

**Think of it as:** `stash` = copy to memory, `stash pop` = paste back.

## 16. Production Branching Strategies

How real engineering teams organize branches. Learn these — you'll be asked in interviews.

### 🥇 GitHub Flow (most common — start here)

```
main (always deployable, protected)
  └── feature/xyz → PR → review → merge → deploy
```

- `main` is always production-ready.
- Every piece of work = short-lived feature branch + PR.
- Simple, perfect for web apps with continuous deployment.

### 🥈 Git Flow (large products, scheduled releases)

```
main          ← production releases only, tagged (v1.0, v1.1)
develop       ← integration branch, always has the latest work
feature/*     ← branch off develop, merge back to develop
release/*     ← stabilize a release, then merge to main + develop
hotfix/*      ← branch off main for urgent production fixes
```

More ceremony, but great for versioned software (mobile apps, desktop, ERP systems).

### 🥉 Trunk-Based Development (Google/Facebook style)

- Everyone commits to `main` (or merges within 24–48h).
- Incomplete features hidden behind **feature flags**.
- Requires excellent CI and test coverage.

### Environment branches (common in startups)

```
main / production   ← what customers use
staging             ← pre-production testing
dev                 ← daily integration
```

Code flows `feature → dev → staging → main` via PRs, each environment auto-deploying its branch.

## 17. Professional Commit Messages

Bad: `fix`, `update`, `changes`, `asdfgh`.
Good: explains **what** and **why** in a structured way.

### Conventional Commits (industry standard)

```
<type>(<scope>): <short summary>

[optional body — what & why, not how]

[optional footer — BREAKING CHANGE, closes #issue]
```

**Types:**

| Type         | Use for                     |
| ------------ | --------------------------- |
| `feat`     | new feature                 |
| `fix`      | bug fix                     |
| `docs`     | documentation only          |
| `style`    | formatting, no logic change |
| `refactor` | code change, no feature/fix |
| `perf`     | performance improvement     |
| `test`     | adding/fixing tests         |
| `chore`    | build, deps, tooling        |

**Examples:**

```bash
git commit -m "feat(auth): add JWT refresh token rotation"
git commit -m "fix(cart): prevent duplicate items on rapid clicks

Rapid add-to-cart clicks fired parallel requests. Debounced the
handler and added a unique constraint at the DB level.

Closes #142"
```

### Multi-line commits (from my notes)

```bash
git commit        # opens editor: line 1 = subject (≤72 chars), blank line, then body
```

### The rules that matter

1. **Imperative mood:** "Add login" not "Added login".
2. **Subject ≤ 72 characters**, no period at the end.
3. **Atomic commits:** one commit = one logical change. Commit early, commit often.
4. **Why over what** in the body — the diff already shows what changed.

## 19. Troubleshooting

Real errors you'll hit, with the exact fixes.

### ❌ `refusing to merge unrelated histories`

Happens when a local repo and remote repo were created independently (e.g. you ran `git init` locally AND created the repo with a README on GitHub).

```bash
git pull origin main --allow-unrelated-histories
# or during merge:
git merge origin/alim-dev --allow-unrelated-histories
git add .
git commit -m "Merge unrelated histories"
git push origin your-branch
```

### ❌ Push rejected / non-fast-forward

Someone pushed while you were working. Your history is behind.

```bash
git pull origin main        # get their work first (merge)
# resolve conflicts if any
git push origin main
```

> ⚠️ Never reach for `git push --force` on shared branches. If you must force-push your OWN branch, use the safer `git push --force-with-lease`.

### ❌ Can't switch branches ("Your local changes would be overwritten")

```bash
git stash                  # park your work
git switch other-branch
# ...later...
git stash pop
```

Or commit the work-in-progress first.

### ❌ Ignored files still getting pushed after adding them to `.gitignore`

They're already tracked — clear the cache (Chapter 5):

```bash
git rm -r --cached . && git add . && git commit -m "Rebuild git cache with .gitignore"
```

### ❌ "I committed to the wrong branch!"

```bash
git log --oneline                 # copy the commit hash
git switch correct-branch
git cherry-pick <commit-hash>     # apply it here
git switch wrong-branch
git reset --hard HEAD~1           # remove it from the wrong branch
```

### ❌ "I need to undo everything and panic!"

```bash
git reflog                        # find the hash of the last good state
git reset --hard <hash>           # time-travel back to it
```

### ❌ Merge a PR from the command line

```bash
git checkout main
git pull origin pull/<PR_number>/merge
git push origin main
```

## 20. Tips & Tricks Master Collection

Everything hard-won from my notes, in one place.

### 🧠 Habits

1. **`git status` is your heartbeat** — run it before and after everything.
2. **Pull before you merge, pull before you push.** Most errors come from skipping this.
3. **Commit early, commit often** — small atomic commits are easy to review, revert, and cherry-pick.
4. **Never commit secrets** (`.env`, API keys). If one slips in, rotate it immediately — deleting the file doesn't remove it from history.
5. **Don't `git add .` blindly** — glance at `git status` first so you don't commit `node_modules` or a stray dump file.

### ⚡ Speed

6. `git log --oneline` for everyday history; `git log --graph --oneline --all` when branches get tangled.
7. `git branch --show-current` inside scripts and when your terminal prompt doesn't show it.
8. Install **Git Graph** in VS Code — `Ctrl+Shift+P` → "Git Graph" → click commits to inspect. Best bug-hunting tool there is.
9. `git diff` (unstaged) and `git diff --staged` (what will go into the next commit) — review before committing.
10. `gh pr create --base main --head dev1 --title "..." --body "..."` — PRs without leaving the terminal.

### 🛟 Safety

11. `git stash` / `git stash pop` — park unfinished work instead of making garbage commits.
12. `git reflog` — the undo history for your undo commands. Almost nothing is truly lost.
13. Prefer `git revert` over `git reset --hard` for anything already pushed.
14. `git clean -fd` deletes untracked files **permanently** — dry-run first with `git clean -nd`.
15. `--force-with-lease` instead of `--force` — it refuses to overwrite if someone else pushed meanwhile.

### 🧹 Hygiene

16. Generate `.gitignore` from [gitignore.io](https://www.toptal.com/developers/gitignore/) — never hand-write it.
17. Added something to `.gitignore` but it keeps coming back? `git rm --cached <file>` (or the full cache rebuild).
18. Delete merged branches: `git branch -d feature/x` and `git push origin --delete feature/x`. A tidy repo is a happy repo.
19. After cloning, `main` is automatic but other branches need `git checkout -b name origin/name`.
20. SSH keys > tokens > passwords. One key/token per machine, passphrases on keys, expirations on tokens.

### 🏢 Team play

21. `main` is sacred: protected branch, PRs only, CI green before merge.
22. Branch names: `feature/x`, `bugfix/y`, `hotfix/z` — your team reads these in lists every day.
23. Keep PRs small and descriptive. Review is a conversation, not a hurdle.
24. Updating your dev branch from another branch: switch to the *receiver*, merge the *source*, then push:
    ```bash
    git switch momen-dev && git merge origin/momen && git push origin momen-dev
    ```
25. Replacing a branch's entire content with another branch:
    ```bash
    git switch momen && git reset --hard hello && git push --set-upstream origin momen
    ```
26. On borrowed machines: temporary token or dedicated SSH key → work → **erase credentials and revoke the token**. Leave no trace.
